# Task: Progress Persistence and Resume

## Description
Implement persistent progress tracking for batch processing operations so that interrupted jobs can be resumed without reprocessing already-completed files. This is critical for handling large file collections where processing may take hours and interruptions are likely.

## Objectives
- Create persistent job state storage for batch operations
- Implement checkpoint mechanism for in-progress batches
- Build resume logic that skips already-processed files
- Add job history with completion statistics
- Support partial result recovery from interrupted jobs

## Acceptance Criteria
- [ ] Job state persisted to SQLite database (file path, status, result hash)
- [ ] Checkpoint written after each file completes processing
- [ ] Resume command: `file-organizer batch resume <job-id>`
- [ ] Skip already-processed files on resume (verified by file hash)
- [ ] Handle files modified since last processing (re-process if changed)
- [ ] Job history viewable: `file-organizer batch history`
- [ ] Cleanup old job records after configurable retention period
- [ ] Atomic state updates (no corruption on crash)
- [ ] Progress percentage calculation across resume operations
- [ ] Unit tests for persistence and resume logic
- [ ] Integration tests simulating interrupted and resumed jobs

## Technical Details
- SQLite database for job state: `~/.file-organizer/jobs.db`
- Schema: jobs (id, start_time, status, total_files, completed_files), job_files (job_id, file_path, file_hash, status, result)
- File hash: SHA-256 for change detection
- Atomic writes via SQLite transactions
- Checkpoint frequency: after each file (default), configurable for batch checkpoints
- Key modules: `services/batch/persistence.py`, `services/batch/checkpoint.py`, `services/batch/resume.py`
