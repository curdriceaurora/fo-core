# Bug #446: CLI Documentation Validation Gap

## Issue
`_command_is_documented()` accepts commands documented in table rows or usage lines as "documented", but `_get_command_section()` (used for required-parameter coverage) only works with markdown headers. This creates a gap where:
- TestAllCommandsDocumented passes (command found in table/usage line)
- TestRequiredArgsDocumented silently skips parameter validation (no header to extract)

## Problem
Two different validation strategies for "documented":
1. **Command presence check**: Accepts table rows, usage lines, headers
2. **Parameter extraction**: Only works with markdown headers

A command can pass "all commands documented" but have zero parameter validation because the section extraction fails silently.

## Solution
Either:
1. **Require headers for all documented commands** (recommended)
   - Update `_command_is_documented()` to use `_get_command_section()`
   - Ensures consistent documentation structure
   
2. **Expand parameter extraction** (complex)
   - Make `TestRequiredArgsDocumented` validate against table/usage-line locations
   - More permissive but harder to maintain

Recommended approach: Option 1

```python
def _command_is_documented(doc_content, command_path):
    """Check whether *command_path* has a dedicated docs section.
    
    A command is considered documented if and only if 
    _get_command_section can find a markdown header for it. 
    This ensures that any command which passes the "all commands 
    documented" test also has an extractable section for 
    required-parameter coverage.
    """
    section = _get_command_section(doc_content, command_path)
    return bool(section)
```
